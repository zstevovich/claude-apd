# v6.8 — Plan-spec consistency gate (structural R-id traceability between spec-card.md and implementation-plan.md)

> Pisano u v6.7.7 sesiji, 2026-05-20. Background discussion record:
> `~/.claude/plans/hajde-da-prodiskutujemo-na-sequential-giraffe.md`
> (convergence + design rationale); ovaj dokument je implementation design.

## Cilj

Zatvoriti **plan-spec ambiguity rupu** — root cause analize iz 2026-05-20 dijaloga: pravi koren defekata nije adversarial dismissal ili builder overrun, već **rasplinut spec → rasplinut plan → builder radi u magli → downstream gateovi rade na širokoj ambiguity surface**.

Trenutni pipeline ima dva mehanička anchor-a:

1. **`spec-card.md` → tests:** `verify-trace` proverava `@trace R*` markere u test fajlovima per R-criterion (postoji od v3.4).
2. **`implementation-plan.md` postojanje:** `pipeline-advance builder` hard-blocks ako fajl ne postoji (postoji od v3.5).

Nepokrivena rupa između ta dva sloja: **plan tehnicki postoji, ali ne mapira eksplicitno na R-kriterije iz spec-a**. Orchestrator može napisati plan koji preskače R3, ili dispatchuje builder sekciju koja ne implementira ništa iz spec-a. Trenutni gate to ne hvata — builder krene, implementira "po osećaju", reviewer/adversarial završe rad pre nego što se ambiguity površi.

Cilj v6.8: forsirati **strukturni R-id consistency check između spec-card.md i implementation-plan.md** preko per-section `**Implements:**` header-a u plan-u, sa bidirekcionalnom verifikacijom (svaki spec R mora biti referenciran u bar jednoj plan sekciji; svaka plan sekcija mora referencirati ≥1 validan R).

## Ključne dizajn odluke

### Granularnost: per-section, ne per-bullet

`implementation-plan.md` se vec deli na sekcije po dispatch boundary (`### Backend`, `### Mobile`, `### Database`). Per-section anchor je prirodan unit — odgovara jednom builder dispatch-u.

Per-bullet anchor (`- file.ext — change description [R1, R3]`) je previse granularan: orchestrator mora donositi mikro-odluke koje R-id ide na koji bullet, što je friction bez signala. Plus bulleti opisuju **fajlove**, ne **funkcionalnost** — R-criteria mapiraju na funkcionalnost.

**Format:**

```
### Backend
**Implements:** R1, R3

- src/api/products.ts — add GET /products endpoint
- src/api/products.test.ts — smoke test + uniqueness check
- src/db/products.sql — products table schema
```

Header `**Implements:** R1, R3` saopštava: "ova sekcija pokriva R1 i R3 iz spec-card.md". Bullets ostaju deskripcija fajlova bez R-id-ova.

### Bidirekcionalna provera

Single direction je nedovoljan:

| Smer | Hvata |
|------|-------|
| Forward (Plan → Spec) | Plan referencira nepostojeci R7 (typo, copy-paste iz druge task-e, scope drift) |
| Reverse (Spec → Plan) | Spec ima R5 koji nije referenciran ni u jednoj plan sekciji (orchestrator zaboravio) |

Bez forward-a, scope drift prolazi tiho. Bez reverse-a, zaboravljeni R-criteria prolaze tiho (BambiProject MR.5 pattern). Oba smera su jeftina za parser (već imamo R-id liste).

### Gate phase: builder advance, pre dispatch-a

Postavlja se na isto mesto kao postojeci hard-block za "implementation-plan.md not found" — `pipeline-advance builder` branch, posle `verify_done "spec"` ali pre BUILDER_RAN check-a. Razlog: plan-spec consistency je preduslov za smislen builder dispatch. Ako plan ne mapira na spec, dispatchovanje builder-a je trošenje tokena.

**Phase ordering u builder branch-u:**

1. spec.done postoji (postojeci check)
2. implementation-plan.md postoji (postojeci check)
3. **plan-spec consistency gate (NOVO v6.8)** ← ovde
4. BUILDER_RAN check (postojeci, v6.7.7 stale-dispatch filter)
5. Planned-agents-dispatched check (postojeci §381-409 logic)
6. Builder cycle cap (postojeci)

### Strict gate-evolution kroz patch lanac — tri-nivoa flag + default-flip

Migration-friendly rollout preko **tri-nivoa `plan_consistency_gate` polja** u spec-card.md (`strict | warn | off`) sa flip-om default vrednosti kroz patch lanac:

- **v6.8.0 — default `warn`, `strict` kao opt-in.** Sve provere emit-uju WARN po default-u, ne BLOCK. Disciplinovani projekti mogu odmah da test-uju strict mode preko eksplicit `plan_consistency_gate: strict` u spec-card.md. Daje empirijski feedback pre v6.8.1 default escalation-a, bez prisilenog migration paina. Postojeci projekti sa mid-pipeline state-om (BambiProject MR-ovi, FiscalFusionAI cycli) ne pucaju retroaktivno.
- **v6.8.1 — default flip na `strict`.** Forward + reverse + section-without-Implements provere postaju BLOCK po default-u. Eksplicit `plan_consistency_gate: warn` ostaje dostupan za soft mode, `off` za skip entirely. Default-flip je single-line change u parser-u; behavior change za one koji nisu postavili eksplicitnu vrednost.

`off` skip-uje sve provere — za eksperimentalne/exploratorne task-e gde plan je deliberately rough draft. Identicna semantika postojecim opt-out-ovima (`adversarial: skip`, `rationale_gate=off`).

## File format spec — `implementation-plan.md` (v6.8)

```markdown
## Implementation Plan: <task-name>

### Files to modify
**Implements:** none

- `path/to/file.ext` — description (1-2 sentences)

### Files to create
**Implements:** none

- `path/to/new-file.ext` — purpose

### Backend
**Implements:** R1, R3

- src/api/products.ts — add GET /products endpoint
- src/api/products.test.ts — smoke + uniqueness test

### Mobile
**Implements:** R2, R4

- mobile/src/screens/Products.tsx — list rendering
- mobile/src/screens/Products.test.tsx — render test

### Documentation
**Implements:** none

- README.md — usage section

### Agents
**Implements:** none

- backend-api
- mobile-rn
- code-reviewer
```

### Format rules

1. **`**Implements:** R<N>[, R<M>, ...]`** — eksplicitan markdown bold prefix, jedna linija, comma-separated R-id-ovi. R-id format: `R` + jedan ili više cifara (case-sensitive, exact match na spec-card.md ID-jeve).
2. **`**Implements:** none`** — explicit declaration za scaffolding sekcije (dokumentacija, refactoring bez novih R, tooling, file-list sekcije, agents lista, notes). Ne broji se u reverse coverage check.
3. **Pozicija:** header se postavlja kao prva non-blank linija ispod `### <section-name>` heading-a. Parser tolera prazne linije pre/posle.
4. **Strict simetricno pravilo — sve sekcije pod `### <name>` MORAJU imati `**Implements:**` header** (ili `none` za scaffolding). Nema rezervisane liste imena; svaka `### Section` deklarise eksplicit. To eliminise specijalne slucajeve, daje konzistentno pravilo orchestrator-u, i pojednostavljuje parser logiku (`Files to modify`, `Files to create`, `Agents`, `Notes` sekcije sve dobijaju `**Implements:** none` u standardnom plan template-u).

### Strict parser rules (`verify-plan-spec`)

1. **Backward compat — early exit.** Ako `spec-card.md` ne postoji ILI nema R-criteria, exit 0 silent. Ako `implementation-plan.md` ne postoji, exit 0 silent (drugi gate already-blocks).
2. **Mode parse.** Procitati `plan_consistency_gate: (strict|warn|off)` iz spec-card.md (case-insensitive, opcionalno leading hyphen/asterisk). Vrednosti:
   - `off` → skip sve provere, exit 0 sa info linijom `verify-plan-spec: opt-out via spec-card.md`.
   - `warn` → sve provere emit-uju WARN, exit 0.
   - `strict` → sve provere emit-uju BLOCK, exit 1 ako bilo koja propadne.
   - Default ako polje nije navedeno: v6.8.0 = `warn`, v6.8.1+ = `strict` (single-line change u parser-u).
3. **Spec R-id set.** Parse `spec-card.md` po istom regex-u kao `verify-trace` (`^\s*-\s+(R[0-9]+)\s*:`). Build set `SPEC_RIDS`.
4. **Plan sections.** Parse `implementation-plan.md` po heading regex-u `^### (.+)$`. Build set `PLAN_SECTIONS` (nema rezervisanih imena — strict simetricno pravilo: svaka `### Section` mora imati `**Implements:**`).
5. **Per-section Implements parse.** Za svaku sekciju, scan body do sledećeg `^###` ili EOF za `^\*\*Implements:\*\* (.+)$`. Extract value (comma-separated R-id list ili `none`).
6. **Forward check (Plan → Spec).** Za svaki R-id u svakom `**Implements:**` redu (skip `none`), verifikuj da postoji u `SPEC_RIDS`. Nepoznati R-id → emit WARN/BLOCK (po mode-u) `Plan section "<name>" references unknown R-id: <id>`.
7. **Reverse check (Spec → Plan).** Za svaki R-id u `SPEC_RIDS`, verifikuj da se pojavljuje u bar jednoj `**Implements:**` liniji. Nepokriveni R-id → emit WARN/BLOCK `Spec R<N> not implemented by any plan section`.
8. **Section-without-Implements check.** Za svaku sekciju koja nema `**Implements:**` header, emit WARN/BLOCK `Plan section "<name>" has no **Implements:** header (add R-ids or set to "none")`.

## Plan migracije — fazno

### Phase 1 — Parser + builder gate (v6.8.0, default `warn` + `strict` opt-in)

**Deliverable:** novi `plugins/apd/bin/core/verify-plan-spec` parser; `pipeline-advance builder` poziva ga posle implementation-plan.md presence check; default mode je `warn` ali korisnik moze da postavi `plan_consistency_gate: strict` u spec-card.md za immediate BLOCK behavior.

1. **Novi binary `plugins/apd/bin/core/verify-plan-spec`:**
   - Header comment + APD style imports (`source resolve-project.sh`, `source style.sh`).
   - Backward compat early exit (rule 1).
   - Mode parse iz spec-card.md (`plan_consistency_gate: strict|warn|off`); default v6.8.0 = `warn`.
   - Spec R-id parse (reuse `verify-trace` regex).
   - Plan section parse (`awk` slice between `^### ` markers).
   - Per-section `**Implements:**` extract (regex match).
   - Tri provere (forward, reverse, section-without-Implements) sa STDOUT WARN/BLOCK linijama po mode-u.
   - Exit codes: 0 (no findings ili `warn` mode sa findings), 1 (`strict` mode sa bilo kojom propadnutom proverom).

2. **`pipeline-advance` builder branch:**

     ```bash

     if ! bash "$SCRIPT_DIR/verify-plan-spec" 2>&1; then
         echo "BLOCKED: plan-spec consistency check failed (see warnings above)." >&2
         echo "  Fix .apd/pipeline/implementation-plan.md per messages, then re-run." >&2
         exit 1
     fi
     ```
   - Skripta sama odlucuje exit code po mode-u; pipeline-advance samo proverava exit code i propagira BLOCK message.

   - Dodati pravilo "Every `### Section` in implementation-plan.md MUST declare which R-criteria it implements via `**Implements:**` header. Use `**Implements:** none` for scaffolding sections (Files to modify, Files to create, Agents, Notes, Documentation)."

## Resolved decisions (2026-05-22 dialogue)

Sledecih 7 odluka je konvergirano u dijalogu posle objektivne benefit/degradation analize:

1. **Syntax:** single-line, comma-separated (`**Implements:** R1, R3, R5`). Per-R rationale, ako bude potreban, ide u plan body bullets ili u dismissal-rationale.md, ne u Implements header.

2. **Scaffolding sekcije:** eksplicitan `**Implements:** none`. Izostavljanje header-a = WARN (signal "orchestrator zaboravio"). Workflow.md eksplicitno: `none` se koristi SAMO za scaffolding, nije escape iz coverage-a (reverse check zatvara gaming pattern bez obzira).

3. **Rezervisana imena sekcija — strict simetricno pravilo:** SVE sekcije pod `### <name>` MORAJU imati `**Implements:**` ili `none`. Eliminise specijalne slucajeve, daje konzistentno pravilo orchestrator-u, pojednostavljuje parser logiku. Standardne file-list / metadata sekcije (`Files to modify`, `Files to create`, `Agents`, `Notes`) u updated workflow.md template-u sve dobijaju `**Implements:** none`.

4. **Escalation timing — fazno sa opt-in od dana 1:** v6.8.0 ship-uje sa default `warn` mode-om ALI `strict` kao opt-in flag (`plan_consistency_gate: strict` u spec-card.md) odmah dostupan. Disciplinovani user-i mogu odmah test-uju strict mode na svojoj task-i. v6.8.1 flip-uje default na `strict`.

5. **Gate phase:** samo builder branch. Post-builder plan-edit je vec scope-drift anti-pattern koji adversarial reviewer i v6.7.7 stale-filter pokrivaju drugim instrumentima. Defense-in-depth nije real-world risk koji opravdava re-check u svakoj fazi.

6. **Opt-out granularnost — tri-nivoa polje:** `plan_consistency_gate: strict|warn|off` u spec-card.md. Default v6.8.0 = `warn`, default v6.8.1 = `strict`. `off` skip-uje sve provere. Tri-nivoa polje prirodno kombinuje sa default-flip kroz patch lanac; orchestrator mental model je konzistentan (`adversarial: max_defects=N|unlimited` ima paralelan multi-value pattern).

7. **Parser:** duplicirati R-id regex iz verify-trace. Refactor u `plugins/apd/bin/lib/parse-spec.sh` kad se pojavi treci consumer (npr. potencijalni `spec-card-validator` u v6.9+). 2 sed linije ne opravdavaju upfront dependency.
