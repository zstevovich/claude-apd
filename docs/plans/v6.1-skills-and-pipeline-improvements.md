## Track A — Skills

### A1. Pushy descriptions (Tier 1, ~30 min)

**Why:** Anthropic skill-creator: *"Currently Claude has a tendency to
'undertrigger' skills... please make the skill descriptions a little
bit pushy."* Our descriptions are 139–202 chars; limit is 1024. 5x
headroom for trigger phrases.

**Targets (8 CC + 7 Codex skills):**

| Skill | Current chars | Target chars | Add trigger phrases |
|---|---|---|---|
| apd-tdd | 167 | ~350 | "implement", "add feature", "fix bug", "write code", "TDD", "test-first" |
| apd-debug | 195 | ~400 | "bug", "fail", "broken", "doesn't work", "error", "crash", "regression" |
| apd-brainstorm | 198 | ~400 | "improve X", "what should we", "thinking about", "options", "vague" |
| apd-finish | 139 | ~350 | "after commit", "push", "PR", "deploy", "next" |
| apd-audit | 185 | ~350 | "audit", "review setup", "is APD configured", "verify framework" |
| apd-github | 181 | ~350 | "GitHub Projects", "issue", "board", "card", "sync" |
| apd-miro | 153 | ~350 | "Miro", "dashboard", "board", "visualize pipeline" |
| apd-setup | 202 | ~400 | "setup APD", "init", "scaffold", "configure", "first run" |

Same treatment for Codex skill mirrors in `plugins/apd/skills/`.

**Spec rules to follow:**
- Third person only (no "I can help...", no "you can use...")
- Both **what** + **when** in same description
- "Triggers on:" phrase listing concrete terms
- Strong verbs ("MANDATORY", "MUST", "ALWAYS") for hard-required skills
- No XML tags, no reserved words ("anthropic", "claude" inside name)

### A2. Concrete examples in 5 skills (Tier 1, ~1h)

**Why:** Anthropic: *"Examples help Claude understand the desired style
and level of detail more clearly than descriptions alone."* Especially
on Sonnet/Haiku.

| Skill | Has examples? | Add |
|---|---|---|
| apd-tdd | ✓ test code example | — |
| apd-brainstorm | ✓ Q&A samples | — |
| apd-debug | ✗ | failing test → 5-why root cause → minimal fix |
| apd-audit | ✗ | FAIL output → concrete recommendation block |
| apd-finish | ✗ | commit message + push/PR/keep decision tree |
| apd-github | ✗ | issue lifecycle (spec → in progress → review → done) |
| apd-miro | ✗ | board update JSON before/after |

Anthropic format: `**Example N:** Input: ... Output: ...` blocks with
2–3 distinct cases per skill.

### A3. Terminology consistency sweep (Tier 1, ~30 min)

**Why:** Anthropic: *"Choose one term and use it throughout the Skill.
Consistency helps Claude understand and follow instructions."*

**Glossary to enforce across all 15 skill files:**

| Pick | Avoid |
|---|---|
| **agent** | subagent, sub-agent |
| **builder agent** (when role-specific) | builder, dev agent |
| **spec-card.md** | spec card, spec-card, the spec |
| **dispatch** (verb) | invoke, call, run-for, kick off |
| **pipeline state** (the .apd/pipeline/ dir) | pipeline phase, pipeline step |
| **pipeline phase** (spec, builder, reviewer, verifier, commit) | pipeline step, pipeline stage |
| **R-criterion / R-criteria** | acceptance criterion, AC, requirement |
| **finding** (adversarial output) | issue, finding, defect — pick consistently per context |

Implementation: one big `sed` per term, then manual review. Apply to
both `skills/` (CC) and `plugins/apd/skills/` (Codex).

### A4. Eval framework (Tier 2, ~1 day)

**Why:** Anthropic: *"Create evaluations BEFORE writing extensive
documentation. This ensures your Skill solves real problems rather
than documenting imagined ones."*

**Deliverable:** per-skill `evals/` folder:

```
skills/apd-tdd/evals/
├── 01-trivial-fix.json       # implement add(a, b) — should write test first
├── 02-bug-fix-no-test.json   # user pastes failing repro — should turn into test
└── 03-edge-case-coverage.json # add validation — should test boundary conditions
```

Each scenario: `{skills, query, files, expected_behavior[]}` per
Anthropic format.

**Plus eval runner script** (`plugins/apd/bin/core/skill-eval`):
- Reads scenario JSON
- Spawns Claude session with skill loaded
- Captures output
- Compares against expected_behavior using LLM-as-judge or rubric
- Emits pass/fail per scenario

Defer to v6.2 if v6.1 chain gets too long. Mark in MEMORY.

### A5. Progressive disclosure for apd-setup (Tier 2, ~2h)

**Why:** apd-setup is 389 lines, approaching Anthropic's "under 500
lines for optimal performance" threshold. Plus it's frequently triggered
(every project init).

**Split:**

```
skills/apd-setup/
├── SKILL.md (overview + workflow, target ~150 lines)
└── reference/
    ├── agent-templates.md (per-stack agent generation rules)
    ├── rules-templates.md (workflow.md + principles.md inserts)
    └── init-checklist.md (verification steps + acceptance criteria)
```

SKILL.md uses Anthropic Pattern 1 (high-level guide with references) —
links to each reference file with "**X**: See [Y](Y)" syntax.

### A6. Codex skill MCP fully-qualified tool names (Tier 3, ~30 min)

**Why:** Anthropic: *"If your Skill uses MCP tools, always use fully
qualified tool names (`ServerName:tool_name`) to avoid 'tool not found'
errors."*

**Audit `plugins/apd/skills/`** (7 Codex skills) for any reference to
APD MCP tools (`apd_ping`, `apd_doctor`, `apd_advance_pipeline`, etc.):
- Wrong: `apd_advance_pipeline`
- Right: `apd:apd_advance_pipeline`

Same for `agents/openai.yaml` files if they reference tools by name.

### A7. Time-sensitive language audit (Tier 3, ~15 min)

**Why:** Anthropic: *"Don't include information that will become
outdated"*.

Grep all skill files for: "recently", "as of 2025", "modern", "new
in vN", "current Codex" — replace with version-anchored or version-free
phrasing.

## Track B — Pipeline gate refinements (v6.1 framework)

### B1. Adversarial pre-flight gate (high impact, ~3h)

**Why:** bambi run #34 evidence: adversarial-reviewer dispatched
03:15:33 (before reviewer.done at 03:18) was wasted — `pipeline-advance
verifier` later refused, forced second dispatch. ~2m 06s lost + cognitive
overhead for orchestrator.

**Fix path 1 — orchestrator instruction (soft):**
- Update `plugins/apd/templates/agent-template.md` (adversarial section)
- Update `plugins/apd/skills/apd-tdd/SKILL.md` Hand-off
- Update `apd_advance_pipeline reviewer` MCP docstring

**Fix path 2 — mechanical block (hard):**
- CC: extend `track-agent` SubagentStart hook — refuse `adversarial-reviewer`
  dispatch when `reviewer.done` not present
- Codex: extend `apd_guard_write` (or new `apd_guard_dispatch` MCP tool)
- Add `.ready-for-adversarial` marker created by `pipeline-advance reviewer`

Ship both — soft path catches well-aligned orchestrators, hard path
catches drift / panic / Codex.

### B2. Severity threshold parameter (high impact, ~2h)

**Why:** Current `pipeline-advance verifier` ignores `ADV_D` count
(advisory by design). Bambi #34 commit-ed with 8 defects open. User
should be able to opt-in to stricter gate.

**Fix:**
- New spec-card field: `adversarial: max_defects=N` (default `unlimited`
  for backward compat — same behavior as today)
- `pipeline-advance verifier` parses spec-card frontmatter; blocks when
  `D > max_defects`. Exit 2 with: `"adversarial summary has D=8,
  spec-card sets max_defects=2 — fix or raise threshold"`
- `apd-brainstorm` skill prompts user for max_defects when generating
  spec-card; recommends:
  - Hotfixes (1–2 R): `max_defects=unlimited`
  - Real tasks (3+ R): `max_defects=0`
  - Complex (5+ R): `max_defects=0` or N with rationale

### B3. Reviewer + adversarial maxTurns bump (medium impact, ~30 min)

**Why:** bambi run #34: 2 maxTurn exhausts (code-reviewer #1 4m 8s,
adversarial-reviewer #2 5m+). Both reviewers underprovisioned for 7-R
features. Memory feedback `feedback-maxturn-counterintuitive-speedup.md`
already validated higher maxTurns wins.

**Fix:**
- `plugins/apd/templates/reviewer-template.md` — bump `maxTurns` from
  current default to ~50–60 (measure from bambi #34 timing)
- `plugins/apd/templates/adversarial-reviewer-template.md` — same bump
  (context-free does NOT mean fewer turns; reviewer still walks full diff)
