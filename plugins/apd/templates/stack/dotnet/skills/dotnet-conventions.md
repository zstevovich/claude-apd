# .NET Conventions

> Generic .NET clean-architecture patterns. This skill is generated by `/apd-setup` when a `.NET` stack is detected. Customize for your project — add your i18n conventions, message bus choice (Wolverine/MediatR/MassTransit), authorization patterns. The defaults below cover patterns common across most production .NET 8/10 services.

## Result Pattern

Use `Result<T>` for ALL business operations. Never throw exceptions for business errors (only for programmer errors like invalid arguments).

```csharp
// CORRECT
public static Result<User> CreateUser(string email, string password)
{
    if (string.IsNullOrWhiteSpace(email))
        return new Error("User.EmailRequired", "user.email_required");
    if (!IsStrongPassword(password))
        return new Error("User.WeakPassword", "user.weak_password");
    return User.Create(email, password);
}

// WRONG — exception for business error
if (string.IsNullOrWhiteSpace(email))
    throw new InvalidOperationException("Email required");
```

Endpoint returns `Results.Ok(result.Value)` on success, `result.ToProblemDetails()` on failure.

## Domain Model — Aggregate Patterns

```csharp
public sealed class EntityName : BaseEntity
{
    private EntityName() { } // Private constructor — EF Core uses it

    public static Result<EntityName> Create(/* parameters */)
    {
        ArgumentException.ThrowIfNullOrWhiteSpace(name);
        ArgumentOutOfRangeException.ThrowIfEqual(id, Guid.Empty);
        // Validation that returns Result<T> on business failure
        // ArgumentException for programmer errors (null/empty contracts)
        return new EntityName { /* setters */ };
    }

    // Private setters — mutations through methods only
    public string Name { get; private set; } = null!;

    // Child collections — encapsulated
    private readonly List<Child> _children = [];
    public IReadOnlyCollection<Child> Children => _children.AsReadOnly();

    public Result<Unit> AddChild(Child child)
    {
        if (_children.Any(c => c.Id == child.Id))
            return new Error("Entity.DuplicateChild", "entity.duplicate_child");
        _children.Add(child);
        return Unit.Value;
    }
}
```

Key principles:
- `sealed` by default unless you need inheritance
- Private constructor for EF Core hydration
- Static `Create` factory enforces invariants
- Private setters — mutations through behavior methods
- Collections exposed as `IReadOnlyCollection<T>` via backing field

## Thin Endpoints — Endpoint ↔ Handler Mapping

Endpoint is a THIN layer — only HTTP-to-handler translation. Business logic belongs in Application/Features handlers, not in the endpoint method.

```csharp
// CORRECT — endpoint maps only
public static async Task<IResult> Handle(UpdateUserCommand cmd, IMessageBus bus)
{
    var result = await bus.InvokeAsync<Result<UserResponse>>(cmd);
    return result.IsSuccess ? Results.Ok(result.Value) : result.ToProblemDetails();
}

// WRONG — business logic in endpoint
public static async Task<IResult> Handle(UpdateUserCommand cmd, IUserRepository repo)
{
    var user = await repo.GetByIdAsync(cmd.UserId);
    if (user is null) return Results.NotFound();
    user.UpdateProfile(cmd.Name);  // business logic — belongs in handler
    await repo.SaveChangesAsync();
    return Results.Ok();
}
```

Why: testability (handlers testable without HTTP), reusability (same handler for HTTP + background + integration tests), separation (endpoint owns transport concerns, handler owns business).

## EF Core Defaults

Common configuration choices for production .NET services:

- **Snake_case naming convention** for tables and columns (PostgreSQL common, applicable to other RDBMS)
- **Soft delete** via `deleted_at` column where applicable; query filters on `DbContext.OnModelCreating`
- **`EnableRetryOnFailure(3, TimeSpan.FromSeconds(2))`** on connection string for transient failure resilience
- **Concurrency tokens** (`xmin` for Postgres, `rowversion` for SQL Server) on entities with concurrent update potential
- **EF Core Design tools** MUST match runtime EF Core version exactly (e.g., both 10.0.0)

```csharp
// In DbContext.OnModelCreating
modelBuilder.Entity<User>()
    .HasQueryFilter(u => u.DeletedAt == null);
modelBuilder.Entity<User>()
    .Property(u => u.Xmin)
    .HasColumnName("xmin")
    .HasColumnType("xid")
    .ValueGeneratedOnAddOrUpdate()
    .IsConcurrencyToken();
```

## FluentValidation — Pitfall to Avoid

FluentValidation runs BEFORE endpoint code executes. URL path parameters (e.g., `/users/{id}`) and JWT-derived claims are NOT yet bound when the validator runs — `id` will be `Guid.Empty`.

Do NOT validate URL-path IDs or JWT-claim values in FluentValidation rules. Validate in the handler instead, where the values are bound.

## Migrations — Discipline

- One migration per logical schema change (don't bundle 5 unrelated tables)
- Down migration MUST work — even for production-only forward migrations
- Migration file names should describe the change: `AddUserSoftDeleteColumn` not `Migration_20260526`
- Test migrations on a fresh DB (full chain) AND on a clone of production data (data preservation)
- Never `dotnet ef database update` against production directly — generate SQL, review, run via deployment

## Git Discipline (APD-specific)

- **NEVER commit, push, or stage from agents** — orchestrator owns commits after review
- Builder agents focus on writing/editing code; orchestrator coordinates commit timing
- This rule is enforced by APD guard-git hook; agents that try to bypass are blocked
