# EF Core Migrations

> Migration discipline for production .NET services using EF Core. Generic patterns — applicable to PostgreSQL (Npgsql), SQL Server, MySQL, SQLite.

## Naming

Migration class names should describe the schema change in PascalCase, NOT the date or migration number:

```
✓ AddUserSoftDeleteColumn
✓ CreateOrderItemsTable
✓ AddIndexOnUserEmail
✓ RenameProductSkuColumn

✗ Migration_20260526
✗ Update1
✗ Changes
```

EF Core auto-prefixes with timestamp (e.g., `20260526123045_AddUserSoftDeleteColumn`); your name needs to describe WHAT changed, not WHEN.

## One Logical Change Per Migration

DO NOT bundle multiple unrelated schema changes into one migration. Each migration should be reviewable as a single unit and rollback-able independently.

```csharp
// CORRECT — one migration per change
AddUserSoftDeleteColumn          // adds users.deleted_at + index + filter
AddOrderRefundedFlag             // adds orders.refunded boolean
RenameProductSkuToBarcode        // schema rename in isolation

// WRONG — bundled
RefactorWeek26                   // adds 4 columns to 3 tables — impossible to review/rollback piece-by-piece
```

## Down Migration MUST Work

The `Down()` method must actually reverse the `Up()` method. Don't leave `Down()` empty or throwing — production rollback depends on it.

```csharp
// CORRECT
protected override void Up(MigrationBuilder migrationBuilder)
{
    migrationBuilder.AddColumn<DateTime?>(
        name: "deleted_at",
        table: "users",
        type: "timestamp with time zone",
        nullable: true);

    migrationBuilder.CreateIndex(
        name: "idx_users_deleted_at",
        table: "users",
        column: "deleted_at",
        filter: "deleted_at IS NULL");
}

protected override void Down(MigrationBuilder migrationBuilder)
{
    migrationBuilder.DropIndex(name: "idx_users_deleted_at", table: "users");
    migrationBuilder.DropColumn(name: "deleted_at", table: "users");
}

// WRONG — irreversible without justification
protected override void Down(MigrationBuilder migrationBuilder)
{
    throw new NotImplementedException("Forward only");
}
```

Even production-only forward migrations should attempt a sensible down — at minimum, dropping new columns and indexes added in Up.

## Testing — Two Chains

Run migrations against TWO datasets before deployment:

1. **Fresh DB chain** — `dotnet ef database drop && dotnet ef database update` should complete the full migration chain from zero on a clean DB
2. **Production-clone chain** — restore a recent production backup, run new migrations, verify (a) no errors, (b) data preserved, (c) query performance not regressed (run smoke queries against new schema)

The fresh chain catches dependency errors between migrations. The clone chain catches data-migration bugs (NULL-to-NOT-NULL columns with no default, type changes on existing data, etc.).

## Production Deployment — Generate SQL, Don't Auto-Apply

NEVER run `dotnet ef database update` against production. Instead:

```bash
# Generate idempotent SQL from current state to target migration
dotnet ef migrations script --idempotent --output deploy/migration_2026-05-26.sql

# Review the SQL — check for table locks, large data backfills, missing indexes
# Run via deployment pipeline (Flyway/Liquibase/raw psql in deploy script)
```

Why: production needs explicit review, change windows, possible online-schema-change tools (gh-ost, pt-online-schema-change) for large tables. Auto-apply hides what's running.

## Concurrency Tokens

Entities with concurrent update potential (multi-user editing, background jobs racing) MUST have a concurrency token:

```csharp
// PostgreSQL — use xmin (system column)
modelBuilder.Entity<Order>()
    .Property(o => o.Xmin)
    .HasColumnName("xmin")
    .HasColumnType("xid")
    .ValueGeneratedOnAddOrUpdate()
    .IsConcurrencyToken();

// SQL Server — use rowversion
modelBuilder.Entity<Order>()
    .Property(o => o.RowVersion)
    .IsRowVersion();
```

Without concurrency token: silent last-write-wins, data loss. With token: `DbUpdateConcurrencyException` on concurrent write — handler can retry or report to user.

## Bypass — Raw SQL for Concurrent Single-Column Updates

For high-frequency single-column updates (lastLoginAt, viewCount, hitCounter) that should NOT trigger concurrency exceptions, use raw SQL that bypasses EF change tracker:

```csharp
await dbContext.Database.ExecuteSqlInterpolatedAsync(
    $"UPDATE users SET last_login_at = NOW() WHERE id = {userId}");
```

Why: pulling the full entity, updating one column, and saving triggers concurrency check on Xmin even when no real conflict exists. Raw UPDATE bypasses it.

## EF Core Version Matching

`Microsoft.EntityFrameworkCore.Design` MUST match runtime EF Core version EXACTLY. Mismatch produces confusing migration errors (e.g., "table already exists" on `Up()`).

```xml
<!-- Both 10.0.0 -->
<PackageReference Include="Microsoft.EntityFrameworkCore" Version="10.0.0" />
<PackageReference Include="Microsoft.EntityFrameworkCore.Design" Version="10.0.0" PrivateAssets="all" />
<PackageReference Include="Npgsql.EntityFrameworkCore.PostgreSQL" Version="10.0.0" />
```

Same applies to provider package (Npgsql/SqlServer/MySql/Sqlite).
